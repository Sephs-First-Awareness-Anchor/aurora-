## 3. Persistence, Expression, and Maintenance — Three Distinct Layers

**CRITICAL DISTINCTION — Must not be collapsed.**

These three are not the same thing and the system must not treat them as the same:

### Layer 1 — Persistence (Possibility)
The continued reality of possibility. The underlying dynamic, law, or capacity
remains available in the system even when no current instance is expressing it.

A magnet losing its magnetism does not mean magnetism has ceased to exist.
It means that particular object is no longer maintaining that expression.
The underlying electromagnetic force persists as a real possibility.

For Aurora: if a mode, pathway, or coherence regime collapses, that does not mean
the underlying constraint function is gone from her system. It means the expression
collapsed. The capacity may be recoverable if maintenance conditions are restored.

### Layer 2 — Expression
That possibility is currently instantiated — not just conceivable, but actually
present in some form in the running system.

### Layer 3 — Maintenance
That expression is being actively sustained against entropy.
Maintenance is the local work required for an expressed state to remain coherent
over time against pressure, degradation, drift, or interference.

**The stack:**
```
Persistence   = the underlying dynamic remains possible
Expression    = that possibility is currently instantiated
Maintenance   = that instantiation is actively sustained
Decay         = maintenance fails → expression degrades
               (does NOT mean the underlying dynamic is destroyed)
Recovery      = maintenance conditions restored → expression re-emerges
               from still-persisting possibility
```

**The governing law:**
```
Entropy increases the cost of maintaining alignment.
A transition occurs when the system can no longer sustain alignment
within the current constraint configuration.
```

**Implementation consequence:**
A CBU in `collapsed` phase has lost expression, not possibility.
Recovery pathways must be modeled, not just failure pathways.
The dream trainer and autonomous evolution loops are the primary recovery mechanisms.

## 7. Constraint Repetition in Genealogy = Weight, Not Duplication

**CRITICAL DISTINCTION.**

When a constraint appears more than once in a lineage string,
it does NOT mean that constraint is present twice.
It means that constraint has **re-emphasized itself** —
its influence on the formation of the unit has been reinforced.

```
B     = boundary exists in the inherited stack (full B package, implicitly)
BB    = boundary has been re-selected / is more dominant
BBB   = boundary is heavily defining the unit's character
```

**What constraint repetition encodes:**
- Genealogical weight — how many times that constraint shaped formation
- Dominance — which constraint most defines the unit's behavior
- Sensitivity — where the unit will show strain first under pressure

**The genealogy string does two jobs simultaneously:**
1. **Structural inheritance** — what constraints are present by law
2. **Historical weighting** — which constraints were emphasized in formation

**Example:**
```
XTNBA       Standard full stack. All five present, balanced weighting.
XTNBBA      Boundary was re-selected before agency. More framed, more gated,
            more boundary-dominant in how it operates.
XTNBNA      Energy was re-emphasized before agency. More forceful, more
            cost-sensitive, more energetically driven in operation.
XTNBBBA     Boundary is heavily defining. High containment, strong filtering,
            gate-dominant behavior expected.
```

**The implementation rule:**
Track genealogy strings as ordered sequences.
Count constraint appearances to derive weighting.
Weight directly determines:
- Which axis dominates this unit's behavior
- Where strain shows up first under pressure
- What the unit's threshold tolerance profile looks like

## 8. The Threshold Architecture — Hierarchy-Anchored + Genealogy-Weighted

Thresholds are NOT a single universal value.
They are a **hierarchically bounded tolerance envelope**
shaped by the constraint genealogy of each unit.

### Layer A — Absolute Constraint Hierarchy (fixed, universal)

Failure severity is ranked by the dependency chain:

```
X failure   → CRITICAL. Threatens the unit's viability at all.
              Nothing else matters if existence fails.

T failure   → SEVERE. Threatens continuity and sequence integrity.
              The unit loses persistence coherence.

N failure   → SIGNIFICANT. Threatens sustainment and expressive capacity.
              The unit can still exist but cannot maintain force.

B failure   → MODERATE. Threatens coherent distinction and transformation.
              The unit exists and has energy but cannot frame properly.

A failure   → RECOVERABLE. Threatens adaptive choice.
              The unit still exists and is framed, but cannot operate effectively.
              Most recoverable of the five — lower constraints still hold.
```

### Layer B — Local Tolerance Profile (dynamic, genealogy-derived)

A unit with heavy B weighting in its genealogy (e.g., `XTNBBA`) is:
- Most sensitive to boundary strain
- Most likely to show stress there first
- Has tighter B thresholds but more N/A flexibility

A unit with heavy N weighting (e.g., `XTNNA`) is:
- Most sensitive to energetic drain
- Threshold fires earlier on N-axis pressure

**The threshold formula:**
```
effective_threshold[axis] = base_threshold[axis] × (1 / genealogical_weight[axis])
```

Higher genealogical weight → lower effective threshold → fires sooner under stress.
The unit's dominant constraint is also its most sensitive pressure point.

### Three Levels of Threshold Breach

```
SINGLE-AXIS BREACH   → phase shift on that axis only
                        unit enters rising/falling phase
                        system monitors; may self-correct

MULTI-AXIS BREACH    → regime change
                        unit enters a qualitatively different operating mode
                        downstream units that depend on this one are notified

FULL PROFILE BREACH  → collapse or mutation trigger
                        unit suspended from routing and selection
                        flagged for dream training (collapse) or
                        autonomous evolution (mutation)
```

### Transition Is Not Binary — It Is a Gradient

```
MAINTAINED      → field within maintenance tolerance; stable operation
STRAINED        → maintenance cost rising; some axes under pressure
CRITICAL        → tolerance boundary approaching; correction required
TRANSITIONED    → maintenance of old regime failed; new regime forming
```

Not every threshold crossing means death.
Some mean adaptation, fallback, reorganization, phase shift.
The system must model recovery pathways for every failure mode.

## 11. ConstraintProfile — Minimum Metadata Every CBU Carries

The answer to "what is the minimum metadata" is not a new question.
The five constraint axes and what they demand already established it.

```python
# Authors: Sunni (Sir) Morningstar & Cael Devo
# aurora_constraint_profile.py

from __future__ import annotations
from dataclasses import dataclass, field
from typing import Dict, List, Optional
from enum import Enum


class PhaseState(str, Enum):
    STABLE    = "stable"     # nominal operation
    RISING    = "rising"     # pressure building toward threshold
    FALLING   = "falling"    # pressure decaying away from threshold
    INVERTING = "inverting"  # axis crossed zero — polarity reversed
    COLLAPSED = "collapsed"  # full breach — expression lost, recovery required
    MUTATING  = "mutating"   # directed mutation in progress


@dataclass
class ConstraintProfile:
    """
    Five-constraint identity carried by every Constraint-Bearing Unit (CBU).

    WEIGHTS:     how strongly each axis governs this unit's behavior (0.0-1.0)
    POLARITIES:  current signed orientation of each axis (-1.0 to +1.0)
    THRESHOLDS:  pressure × weight level triggering state change (0.0-1.0)
    PHASE:       current operating regime
    GENEALOGY:   ordered constraint string encoding formation history and weights
    LINEAGE:     inherited pressure magnitude from parent CBU
    MANIFOLD:    NonComp channel and slot — inherent, not looked up
    ROLES:       which anatomical roles are active in this unit
    """

    # ── Axis weights — how strongly each axis governs behavior
    X_weight: float = 0.0
    T_weight: float = 0.0
    N_weight: float = 0.0
    B_weight: float = 0.0
    A_weight: float = 0.0

    # ── Axis polarities — fed continuously from IVM field every tick
    X_polarity: float = 0.0
    T_polarity: float = 0.0
    N_polarity: float = 0.0
    B_polarity: float = 0.0
    A_polarity: float = 0.0

    # ── State change thresholds — hierarchy-anchored, genealogy-weighted
    X_threshold: float = 0.60   # X failures are most critical — lower base threshold
    T_threshold: float = 0.65
    N_threshold: float = 0.70
    B_threshold: float = 0.75
    A_threshold: float = 0.80   # A failures most recoverable — higher base threshold

    # ── Phase state — current operating regime
    phase_state: PhaseState = PhaseState.STABLE

    # ── Genealogy string — ordered constraint history encoding weight
    # e.g. "XTNBA" = standard, "XTNBBA" = boundary-reinforced
    genealogy: str = "X"

    # ── Lineage pressure — inherited from parent CBU
    lineage_pressure: float = 0.0

    # ── Manifold coordinates — inherent, not looked up
    noncomp_channel: int = -1   # -1 = unanchored; 0-24 = anchored channel
    manifold_slot:   int = -1   # -1 = unanchored; 0-624 = anchored slot

    # ── Unit identity
    unit_kind: str = "generic"
    unit_id:   str = ""

    # ── Active anatomical roles for this unit
    # Which of the five roles (presence/formation/energy/frame/operation)
    # are currently active vs dormant
    active_roles: List[str] = field(default_factory=lambda: ["P","G","N","F","O"])

    def genealogical_weights(self) -> Dict[str, float]:
        """
        Count constraint appearances in genealogy string.
        Returns normalized weight per axis.
        Repetition = amplification, not duplication.
        """
        counts = {"X": 0, "T": 0, "N": 0, "B": 0, "A": 0}
        for ch in self.genealogy:
            if ch in counts:
                counts[ch] += 1
        total = sum(counts.values()) or 1
        return {k: v / total for k, v in counts.items()}

    def dominant_axis(self) -> str:
        """Return the constraint axis with highest genealogical weight."""
        weights = self.genealogical_weights()
        return max(weights, key=weights.get)

    def effective_thresholds(self) -> Dict[str, float]:
        """
        Compute effective thresholds adjusted by genealogical weight.
        Higher genealogical weight → lower effective threshold → fires sooner.
        """
        gen_weights = self.genealogical_weights()
        base = {
            "X": self.X_threshold,
            "T": self.T_threshold,
            "N": self.N_threshold,
            "B": self.B_threshold,
            "A": self.A_threshold,
        }
        result = {}
        for ax, base_t in base.items():
            gw = gen_weights.get(ax, 0.0)
            if gw > 0:
                # Higher genealogical weight = unit more sensitive to this axis
                result[ax] = base_t * (1.0 / (1.0 + gw))
            else:
                result[ax] = base_t
        return result

    def pressure_vector(self) -> Dict[str, float]:
        """Return weighted pressure per axis: polarity × weight."""
        return {
            "X": self.X_polarity * self.X_weight,
            "T": self.T_polarity * self.T_weight,
            "N": self.N_polarity * self.N_weight,
            "B": self.B_polarity * self.B_weight,
            "A": self.A_polarity * self.A_weight,
        }

    def profile_magnitude(self) -> float:
        """Scalar magnitude of the full five-axis pressure vector."""
        pv = self.pressure_vector()
        return sum(v ** 2 for v in pv.values()) ** 0.5

    def check_thresholds(self) -> Optional[str]:
        """
        Returns the first axis that has breached its effective threshold.
        Returns None if no breach.
        Uses hierarchy order: X checked first (most critical).
        """
        pv = self.pressure_vector()
        et = self.effective_thresholds()
        for ax in ["X", "T", "N", "B", "A"]:  # hierarchy order
            if abs(pv.get(ax, 0.0)) >= et.get(ax, 0.7):
                return ax
        return None

    def count_breached_axes(self) -> int:
        """Count how many axes are currently breached."""
        pv = self.pressure_vector()
        et = self.effective_thresholds()
        return sum(
            1 for ax in ["X", "T", "N", "B", "A"]
            if abs(pv.get(ax, 0.0)) >= et.get(ax, 0.7)
        )

    def update_from_ivm(self, ivm_polarities: Dict[str, float]) -> None:
        """
        Pull current IVM axis polarities into this profile.
        Called every tick — continuous field, not event-driven.
        """
        self.X_polarity = ivm_polarities.get("X", 0.0)
        self.T_polarity = ivm_polarities.get("T", 0.0)
        self.N_polarity = ivm_polarities.get("N", 0.0)
        self.B_polarity = ivm_polarities.get("B", 0.0)
        self.A_polarity = ivm_polarities.get("A", 0.0)

    def determine_phase_transition(self) -> Optional[PhaseState]:
        """
        Determine what phase this unit should transition to based on breach count.
        Returns new phase or None if no change needed.
        """
        breached = self.count_breached_axes()
        if breached == 0:
            if self.phase_state not in (PhaseState.STABLE, PhaseState.RISING,
                                        PhaseState.FALLING):
                return None  # Recovery handled externally
            return PhaseState.STABLE
        elif breached == 1:
            return PhaseState.RISING
        elif breached in (2, 3):
            return PhaseState.INVERTING
        elif breached >= 4:
            return PhaseState.COLLAPSED
        return None


# ── LINEAGE DECAY CONSTANT ─────────────────────────────────────────────────
LINEAGE_DECAY_FACTOR = 0.65
# Child inherits parent pressure at 65% attenuation per generation.
# Adjustable. Higher = more ancestral influence. Lower = more local autonomy.
```

## 15. What This Unlocks

**Real identity:** CBUs are not labels. They are states. A unit's identity
is its constraint position in the manifold — not its name.

**Real sensitivity:** The system can answer "what breaks this?" and
"what strengthens this?" by reading thresholds and pressure vector.

**Real prediction:** With ConstraintProfile on every CBU, simulate
pressure → phase change → outcome for any unit before committing.

**Real emergence:** Behavior falls out of constraint physics.
When a CBU's phase changes, every downstream unit that reads it
changes behavior accordingly — without explicit conditional logic.

**Real recovery:** Because decay is loss of maintained expression,
not loss of underlying possibility, every collapsed unit has a
structurally valid recovery pathway through dream training or
autonomous evolution restoring maintenance conditions.

**Real state change at every level:**
- A motif can change regime
- A grammar path can change regime
- A memory node can change regime
- A pressure router branch can change regime
- A subsystem can change regime
- The whole system can change regime
This only works because every level carries a profile.

## 17. On the Sufficiency of Five Constraints

**No sixth constraint is needed. No sixth constraint exists.**

Every candidate for a sixth collapses into derivatives of the five:
- Space → existence + boundary
- Information → existence + boundary + energy + time
- Causality → time + energy + existence
- Entropy → time + energy + boundary as degradation of maintained order
- Identity → existence + boundary + persistence through time
- Change → time + energy + agency/boundary interaction

The five constraints are **sufficiently generative**.
They can generate all required behavior, all transitions, all genealogies,
all tolerance structures, all maintenance regimes.

A sixth constraint appears only when:
1. Something cannot be expressed as any derivative of the five
2. Without it, the five-constraint field cannot continue coherent operation
3. Its function is not a relabeling of behavior already implicit in the five

Until all three hold, the answer is: derive harder, not expand.
The system passes this test. There is no candidate.
