# Aurora Operator Doctrine (Canonical)

## Purpose
This document is the operator-level doctrine for Aurora runtime evolution. It is normative for CLI operators, contributors, and automation.

## Core Doctrine
1. Intelligence is the path of least resistance.
2. Emergence is required. Do not hardcode final behavior policies as explicit response scripts.
3. Steering is parameter-level, not answer-level.
4. Conflict and pressure are lawful training gradients, not arbitrary noise.
5. Persistence is mandatory: saved evolution state is a valid starting point for later runs.

## Non-Negotiable Engineering Rules
1. No manual behavior scripting to force specific conversational outputs.
2. Change gradients, thresholds, coupling economics, or sampling pressure only.
3. Preserve lineage traceability for all promoted links/abilities/couplings.
4. Keep save-gate review before persistence when alignment is uncertain.
5. Record run diffs and evolution deltas for every major run.

## Evolution Steering Policy
1. Prefer interventions in this order:
   - pressure/cost scaling
   - conflict curriculum intensity/cadence
   - plateau detection sensitivity
   - coupling validation representation thresholds
2. Avoid direct trait injection unless required for system safety.
3. Treat representation checks as validation coverage, not pass/fail grading.

## Economic Law (Current)
1. Formation complexity can increase with depth/structure.
2. Persistence maintenance becomes cheaper with depth after promotion.
3. Persistence tax is applied per tick and modeled against scale-time dynamics.
4. Break-even in ticks must remain observable for each coupling.

## Operational Principle
If a desired behavior is missing, adjust the environment pressures and coupling economics; do not author the behavior directly.

## Scope
This doctrine applies to:
- `aurora_runtime.py`
- `constraint_genealogy.py`
- `aurora_cost_diff_score.py`
- supporting operator tooling and run scripts.
